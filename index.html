Styleguide and CSS Code Standards

Learning A-Z

Problem

After 10 years of Learning A-Z growing fast, adding new features, and acquiring new products, the UI and CSS codebase were fragile and unruly with files thousands of lines long and full of repetition, inconsistencies, and tangled spaghetti code that was clearly reflected in the UI. After some initial team meetings to kick off the project, I got to work putting together a new CSS codebase and styleguide.

Process

Plan & Implement

Initially, the styleguide was built based on SMACSS principles and included a grid system and generic UI components and helpers. After some revisions with the team, the new CSS began being introduced into new projects. Then came the painstaking process of updating the legacy code.

My process was to first take stock of what exists on the sites (buttons, for example)




Next I needed to find a unified approach that would work for all scenarios without requiring controversial change that would lead to a longer approval process. For example, if there were 3 green buttons, I would pick a good green and use it consistently everywhere, but wouldn\'t change green to orange without good reason. Once an approach was approved, I wrote HTML/CSS following new guidelines in the styleguide, and implemented it page by page.



Once everything was cleaned up and consistent, it was much easier to make more dramatic change moving forward, such as removing icons or changing colors.

Evaluate & Pivot

As changes were rolled out, one problem that surfaced was the over-use of helpers. The same complicated patterns of helper classes were being repeated over and over in a huge number of files. The need for a more module-based approach became clear, leading us to adopt BEM, slightly modified to suit work we had already done with the looser SMACSS method. I updated the styleguide with clear definitions for the new modular approach.



Rebranding

As changes continued to evolve and roll out, the parent site was rebranded, giving a perfect opportunity to establish clean, consistent visual standards across all product sites, starting with the site headers.

With each new project and iteration of the styleguide, the process and interface continue to improve. The stylesheets are far more manageable, and the look of the sites continues to become more clean, modern, and consistent.

File Cabinet and Custom Assignments

Learning A-Z

My Assignment

On the Learning A-Z family of sites, teachers can save books and other resources inside of a File Cabinet for future use. They can also assign books for students to complete online.

I was assigned two new projects related to the File Cabinet and Assignments:

  1. Redesign the File Cabinet, including new functionality for teachers to share their collections
  2. Enhance the Assignments feature to Design a way for teachers to create assignments with multiple books, and save them for future use

I knew the features that needed to be added, but didn't know the 'why'. What types of teachers was this for? What are some representative scenarios for how they might use this feature? Nonetheless, I was instructed to move ahead, and took on the challenge from more of an architectural standpoint.

Analysis

When the features were described to me, it sounded like 2 versions of the same feature. I wrote up a simple diagram to think this through and demonstrate my point.

Both features were ways for teachers to collect and organize resources. I suggested one unified project so that teachers aren't collecting resources in two different places.

Next, I had to think about what sort of UI might support flows for browsing, collecting, saving, assigning, and sharing. The process sounded a lot like the familiar ecommerce experiences of finding items, adding them to wishlists and shopping carts, and completing purchases. I did some more diagramming to think this through.

After demonstrating the strong correlation, shopping cart experiences became a big source of reference and inspiration for the File Cabinet and Assignments.

Information Architecture

Before beginning any layouts, I needed to map out the features to better understand the requirements, and how they fit together.

First, I started with the known sitemap, and added the necessary interactions for each page.

Then, I reorganized the map by the sequences of actions users can take.

From this process, 4 project components emerged:

  • File Cabinet
  • File Cabinet Preview
  • Assignment Process
  • Assignment Management

Wireframes

The File Cabinet was redesigned using well-established patterns of digital file management with simplicity as a top priority.

Before creating wireframes, I loosely outlined the File Cabinet information architecture and navigation

Next came sketches.

At this stage, I accepted many assumptions that were made about what would be most important to the majority of teachers. These assumptions came from design and product leadership in the organization.

Mockups

Once the design lead was comfortable with the sketches, final mockups were created.

Iteration

At this point, the numerous assumptions and lack of research and clear use cases began to catch up to us. Feedback from the product team was that it needed to be easier to assign a collection of resources on the spot. Saving this collection for later was a secondary priority. Once again, the 'why' was not understood. It was explained simply as a feature the market was asking for. I moved forward with the assumption that this was correct, and mockups were reworked to satisfy this new requirement.

A ribbon was used to allow teachers to actively see their collection being built, and the button labeling was changed so that it didn't feel like a feature primarily for saving collections.

A similar design process was followed to design screens for creating and managing assignments, and monitoring assignment progress. Limitations based on the existing information architecture and technical limitations were also taken into consideration. The final flow of Assignments screens was like this:

Development

During the design phase, it became clear that this project was a large and complex one, more than originally thought.

During development, new requirements and feedback continued to come to light, leading to constant adjustments to mockups, and frustrating rework for the engineers. Our habit of moving forward with designs without challenging assumptions or gathering clear requirements, personas, and use cases really started to effect the project.

To help organize the chaos, I documented each mockup and requirement, providing a single source of truth to maintain alignment among design, engineering, QA, and product teams as much as possible, and to track changes and progress as they occurred.

Outcome

In the end, teachers were able to assign groups of resources with just a few clicks, easily view those collections in the File Cabinet, and monitor progress on the Assignments page, so for the company, the project was considered a success. However, there was still much work to be done to tie these features together. But many steps were taken in the right direction, resulting in a more useful experience for teachers.

Throughout the project, I made an effort to understand exactly what the priority was - what problems were we trying to solve? What were the key scenarios we wanted to design for? Were our users primarily elementary teachers, or reading specialists? Unfortunately I failed to collect this information, and relied on assumptions and heresay to make decisions and move forward. This project was clear evidence that skipping steps and making assumptions hurts everyone, including our users. I was proud of my ability to step up and help organize the project, but learned the importance of advocating more strongly for design thinking.

Next Gen Writing

Learning A-Z

My Assignment

After obtaining new leadership, merging with acquisitions, starting new scrum agile processes, and reacting to changes to schooling after the 2020 pandemic, Learning A-Z set out to introduce two new "Next Gen" products. Both projects shared similarities so it was a team effort. I was assigned to portions of both, but primarily one, nicknamed Next Gen Writing.

The project ask was massive, and very little direction given. This provided an opportunity to take initiative and try work I normally did not normally do.

Product Definition

After initial meetings with the new scrum team, everyone was struggling to wrap our minds around what we were trying to create.

My first contribution was putting together a simple sitemap to help get everyone on the same page.

This evolved into a more thorough sitemap as our understanding of the product developed.

Our project manager and scrum master introduced us to the concept of story mapping. I worked closely with her and the product owner to fill out the storymap, which became a sort of hybrid between a storymap and a flow.

Product Ecosystem

One thing the team was still struggling with was understanding our existing product line and how the new products would fit in. The team responded very well to my sitemaps, so I created more maps to demonstrate how the new products compared to the existing ones.

The most exciting part of this side-project was when I was able to show a map to visualize where the team ultimately wanted to go - a single Learning A-Z portal where you can access all of your tools for learning, rather than a series of similar but disconnected experiences. These maps excited the team and allowed us to align, visualize, and communicate this shared goal.

Personas

As different ideas were developing and a tight deadline approaching, the team realized a need to prioritize and focus. Developing personas helped us stay on track.

We didn't have time or resources for much research, but our product owner had results from a past survey with writing educators that I was able to pull data from. Using the surveys, I found patterns, discussed my findings with the team, and put the information together in nicely packaged personas to represent who we were creating this product for, and what our priorities needed to be.

UI Patterns

With our Next Gen products, we wanted to set the new standard for our company moving forward. Our existing products were somewhat of a mess. But a lot of redesign iniatives were mid-flight and contained work that would benefit our new products. To ensure valuable work wasn't lost, I put together wireframes to show what our existing patterns and templates look like with all of the legacy mess stripped away, and how Next Gen Writing would look like using them. This also helped give the team a shared understanding of what we meant when we talked about certain pages like the homepage, category page, detail page, or reports.

Prototypes and User Testing

At this point, we needed to get some questions answered, and gut check many assumptions with user testing. The initial testing was focused on the big picture - would teachers understand the product? Would they want to use it? Basically - were we building the right product?

Because this line of questioning was very product-focused, the product owner took ownership of this round of testing. We were very pleased to find that teachers immediately recognized the lesson structure, and how the digital tools were intended to be used. They invisioned using it in their classroom and were excited by what they saw. We wouldn't know for sure until teachers truly tried it out with their class, but the initial feedback was very encouraging.

Next, the UX team completed more detailed testing to narrow down some of the UI decisions we needed to make. I provided prototypes and questions, and another UX team member completed testing for this project alongside the other Next Gen product that was being developed. This testing was surrounding issues like navigation, information architecture, and the hierarchy of information. A few quick tests were very helpful in refining details we were waffling on for our beta version of the Next Gen products, and helping us find blind spots. Fortunately, no major issues were found.

Development

One of my primary focuses with the Next Gen projects was maintaining alignment across teams. This included the work shown above, a lot of ongoing work to coordinate with the other Next Gen product team, and close collaboration with our engineers. As the deadline for our beta release kept getting closer, we needed to understand what tools would be used to build the site, which features were realistic for our timeline, and which features needed to be simplified or adjust to keep them in scope.

The engineering team had decided to start fresh with the Next Gen sites rather than re-using old problematic code built on AngularJS. The new products would be built with an updated version of Angular using Angular Material. Time was going to be extremely tight for development, so after carefully studying Angular Material, the mockups were adjusted to match out-of-the-box Angular Material. This didn't look ideal, but gave a very clear pictures of what could be accomplished within the timeframe we had. Creating a subpar page layout was much preferred over setting up the team to rush and write hundreds of lines of spaghetti styles which would land us in the same mess we were in with our legacy code. This helped ensure that everyone on the team understood what we were doing, and why.

Conclusion

As the beta design phase wrapped up and development began, I left Learning A-Z for a position with another company. As of this writing (7/22), Next Gen Writing still hasn't been launched, so I can't share any final results.

This project was a great example of teamwork, and following process. Instead of thinking to myself "if only I could talk to so and so... if only we would stick to a process... if only..." we did it right. With our project manager advocating strongly for scrum processes, and a new sprint team, and a team of people willing and eager to learn better ways of doing things and wanting to make an awesome product, I was able to tackle the UX in the right way - by understanding the background, working closely with the product team, content editors, instructional designers, developers, and other UX designers. It was a very stressful time, but very rewarding to put in effort to do things right, and see it start to pay off.
